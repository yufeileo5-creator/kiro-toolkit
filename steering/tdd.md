# TDD 红绿重构铁律 (Test-Driven Development)

> 迁移自 `skills/_on-demand/test-driven-development/SKILL.md`。通过 `#tdd` 手动引用。

## 铁律

没有失败的测试，就不允许写生产代码。先写代码再补测试？删掉，重来。

## Red-Green-Refactor 循环

### RED — 写失败测试
- 一个行为，一个测试
- 清晰命名描述行为
- 用真实代码，非必要不 mock

### 验证 RED
运行测试，确认：失败（不是报错）、失败原因符合预期、因功能缺失而失败

### GREEN — 最小代码
写最简单的代码让测试通过。不加额外功能，不重构其他代码。

### 验证 GREEN
运行测试，确认：测试通过、其他测试也通过、输出干净

### REFACTOR — 清理
去重、改名、提取辅助函数。保持测试绿色。不加新行为。

## 常见合理化借口

| 借口 | 现实 |
|------|------|
| "太简单不用测" | 简单代码也会坏，测试只要 30 秒 |
| "之后补测试" | 测试立即通过什么都证明不了 |
| "删掉 X 小时的工作太浪费" | 沉没成本谬误，保留不可信代码才是技术债 |
| "TDD 太教条" | TDD 就是务实：比生产环境调试快 |
| "手动测试更快" | 手动测试不能证明边界情况，每次改动都要重测 |

## Bug 修复流程

1. 写失败测试复现 bug
2. 遵循 TDD 循环修复
3. 测试证明修复有效并防止回归

永远不要在没有测试的情况下修 bug。
