# 系统化调试流程 (Systematic Debugging)

> 迁移自 `systematic-debugging` 技能。通过 `#debugging` 在聊天中手动引用。

核心原则：**没有根因调查，不允许提出修复方案。**

## 何时使用

任何技术问题：测试失败、Bug、异常行为、性能问题、构建失败、集成问题。
尤其是：时间紧迫时、"一个快速修复"看起来很明显时、已经尝试过多次修复时。

## 四阶段流程

### 阶段 1：根因调查

在尝试任何修复之前：

1. 仔细阅读错误信息 — 不要跳过警告，完整阅读堆栈跟踪，记录行号和错误码
2. 稳定复现 — 确定精确步骤，是否每次都发生
3. 检查近期变更 — git diff、最近提交、新依赖、配置变更
4. 多组件系统中收集证据 — 在每个组件边界添加诊断日志，运行一次收集证据，确定哪个层级出问题
5. 追踪数据流 — 坏值从哪里来？谁用坏值调用了这里？一直追溯到源头

### 阶段 2：模式分析

1. 在同一代码库中找到类似的正常工作代码
2. 逐项对比差异，不要假设"那个不重要"
3. 理解依赖关系和前置假设

### 阶段 3：假设验证

1. 形成单一假设："我认为 X 是根因，因为 Y"
2. 做最小变更测试，一次只改一个变量
3. 有效 → 阶段 4；无效 → 形成新假设，不要在上面叠加更多修复

### 阶段 4：实施修复

1. 创建失败测试用例
2. 实施单一修复，只针对根因
3. 验证修复：测试通过？没有破坏其他测试？
4. 如果连续 3 次修复失败 → 停下来质疑架构，与用户讨论

## 危险信号 — 立即停下来

如果你发现自己在想：
- "先快速修一下，以后再调查"
- "试试改 X 看看行不行"
- "同时改多个地方，跑测试"
- "跳过测试，手动验证"
- "我不完全理解但这可能有用"

这些都意味着：停下来，回到阶段 1。
