# 切片执行工作流 (Slice Execution)

> 迁移自反重力体系的 PLANS.md 切片化管理机制。通过 `#slice-execution` 手动引用。
> 将大型功能拆解为独立切片，跨对话持久化追踪进度。

## 核心概念

- **切片 (Slice)**：一个独立的功能单元，对应 `docs/plans/FEAT-xxx.md`
- **PLANS.md**：切片总索引，跨对话可见的进度看板
- **原子任务**：每个切片内的最小执行单元，不超过 3 个文件修改

## 切片管理规则

### 1. 切片创建
- 复杂功能必须先在 `docs/plans/` 创建独立切片文件（如 `FEAT-001-用户认证.md`）
- PLANS.md 只作总索引，不存放详细内容
- 切片文件必须包含：目标、架构决策、任务清单、执行跟踪面板

### 2. 原子任务拆解
- 严禁写粗粒度大型任务包
- 每个 Checkbox 必须拆解到单一组件级别（不超过 3 个文件修改）
- 执行时戴上"认知遮罩"，只关注当前原子任务，禁止盲目扫描无关模块

### 3. 微粒度存档
- 每完成一个原子任务并通过验证后，立即执行：
  ```bash
  git add . && git commit -m "WIP: [切片名-任务名]"
  ```
- 这是随时可回滚的时空锚点

### 4. 收尾反写 (Write-back)
- 功能交付或执行 `#handoff-workflow` 前，必须将完成状态（`[x]`）和执行结论同步反写回切片文件
- 禁止出现「参见对话 artifact」的悬空引用

### 5. Session Hydration（跨对话重建）
- 新对话启动后，如果检测到 PLANS.md 中有活跃切片：
  1. 第一步必须读取对应切片文件，重建实施计划
  2. 复述对当前设计方案和约束的理解
  3. 用户确认后方可编码
- 严禁直接开始编码而不先重建上下文

### 6. 异脑交接 (Cross-Agent ADR Handoff)
- 接手涉及其他模块的新 Feature 时，第一步必须强制加载 `docs/adr/` 决策记录
- 实现 AI 到 AI 的契约交接，防止推翻已有架构决策

## PLANS.md 格式

```markdown
# PLANS.md — 持久化设计文档

## 活跃计划切片 (Active Slices)
- [FEAT-001: 功能名称](docs/plans/FEAT-001.md) Status: Doing
- [FEAT-002: 功能名称](docs/plans/FEAT-002.md) Status: Todo

## 归档库 (Archived Slices)
- [FEAT-000: 已完成功能] Status: Done
```

## 切片文件格式

```markdown
# FEAT-xxx: 功能名称

## 目标
简述要做什么、为什么做。

## 架构决策
关键技术选型和否决方案。

## 任务清单
- [ ] 1. 原子任务（≤3 个文件）
- [ ] 2. 原子任务
- [x] 3. 已完成任务

## 执行跟踪面板
| 任务 | 状态 | 完成时间 | 备注 |
|------|------|---------|------|
| 任务1 | ✅ | 2026-04-06 | |
```

## 与 Kiro Spec 的协作

Kiro 原生 Spec 负责结构化流程（需求→设计→任务），本工作流负责：
- 跨对话的进度持久化（PLANS.md）
- 微粒度存档（WIP commits）
- Session Hydration（新对话重建上下文）
- 收尾反写（保持双向数据一致）
