# 僵尸代码清理工作流 (Dead Code Sweeper)

> 迁移自 `dead-code-sweeper` 技能。通过 `#dead-code-sweeper` 在聊天中手动引用。

## 核心原则

绝对禁止在开发新功能途中"顺手"删除看似无用的代码。清理必须作为独立任务执行。

开发中发现僵尸代码时：
1. 不动手删除
2. 用 `// TODO(dead-code): 疑似未使用` 标记
3. 继续完成当前功能
4. 功能完成后再单独发起清理

## 四阶段流程

### 阶段 1：全局扫描（不修改任何代码）

1. 静态依赖分析（如有 knip：`npm run find-deadcode`）
2. 检查 `.dsp/` 图谱中入度为 0 的孤岛节点
3. 全局文本搜索防误杀（排除 `node_modules`、`dist`），防止动态导入漏判

### 阶段 2：制定清理计划（等待用户批准）

整理嫌疑清单：
- 高确信度（可安全删除）：完全孤立的文件或死函数
- 疑似/需确认（动态风险）：可能通过反射或动态路由引用
- 陈旧配置/依赖：`package.json` 中多余的依赖

提交计划，等待用户确认。

### 阶段 3：原子化切除

- 从末端（叶子节点）开始，自下而上清理
- 每次清理独立提交，不与新功能混在一个 commit

### 阶段 4：验证与同步

- 运行类型检查和测试，确认无误杀
- 如有报错，立刻回退并排查
- 同步更新 `.dsp/` 图谱（如使用）
