# 计划模式工作流 (Plan Mode v2)

> 迁移自 GitHub 最新版 `workflows/plan.md`。通过 `#plan-workflow` 在聊天中手动引用。

## 中断与恢复铁律

- 用户中途切换话题 → 本工作流立即挂起
- 只有用户显式说"继续计划"等恢复指令时才继续
- 恢复时先确认："之前在步骤 X 暂停了，需要继续吗？"

## 阶段一：规划

### 1. 需求澄清与上下文检索
- 仔细阅读需求，有歧义立即提问
- 查阅现有文档和历史决策，严禁从零开始分析
- 阅读 README.md、架构文档及相关设计文档
- 检查是否存在交接文档（`.kiro/context/handoff.md`），如有则继承其中的设计决策和否决方案
- 交接文档中的「用户偏好与隐式约定」等同于用户显式指令，必须遵守

### 2. 方案设计（implementation_plan.md）
- 历史决策防撞墙：确认新设计不违背已有架构契约，如有冲突必须显式标注并等待用户裁决
- 先分析并复用项目中已有的类似模式或工具函数
- 实施计划必须包含：
  - 目标与意图
  - 架构合规声明
  - 变更影响范围（修改文件、间接影响、需同步更新的测试）
  - 详细修改清单（按依赖顺序）
  - 新依赖/模块准入说明
  - 风险与性能评估（`⚠️ BREAKING CHANGE` 标注）
  - 至少一张 Mermaid 可视化图表

### 3. 任务拆解（task.md）
- 拆解为细粒度任务节点，每个任务对应一个原子的、可编译的代码状态

### 4. 用户审批（强制阻塞）
- 提交计划供用户审核
- 在用户明确批准前，严禁执行任何代码修改

## 阶段二：执行

获得批准后，按任务步骤严格执行：
- 测试证伪原则：涉及测试时，先造出明确的红灯报错作为证伪基础
- 所有修改必须完整可运行，严禁占位符
- 强制类型定义，禁用 `any`
- 核心函数 ≤ 40 行，文件 ≤ 300 行
- Mock 隔离：超 15 行的测试数据抽离至 `__fixtures__/`
- 微粒度存档：每完成一个原子任务并编译通过后，主动 `git commit -m "WIP: xxx"`
- 每完成一步同步进度

## 阶段三：验证

- 执行 lint/类型检查，确认无新增错误
- 执行测试验证回归（巨量输出重定向至临时文件，提炼核心错误）
- 记录变更总结、遇到的坑和根因分析
- 如涉及新模块或核心 API 变更：创建 ADR、同步 DSP 图谱、更新交接文档
