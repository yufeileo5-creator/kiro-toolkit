# 上下文交接工作流 (Context Handoff v2)

> 迁移自 GitHub 最新版 `workflows/handoff.md`。通过 `#handoff-workflow` 在聊天中手动引用。
> 核心目标：不仅传递「进度快照」，更传递「推理链」——确保新 AI 知道为什么这样做、否决了什么、约定了什么。

## 阶段 1：深度上下文与状态审计

### 1.1 工作区状态
- `git status` + `git diff --stat` 读取未提交变更
- 检查 `.dsp/` 图谱是否与代码最新状态对齐（如存在）

### 1.2 任务进度
- 读取实施计划和任务清单（如存在）
- 读取 `CHANGELOG.md` 最近条目

### 1.3 质量与防漂移
- 执行测试命令，记录通过率、失败用例和 Warning
- 如有 lint 工具，执行并记录输出

### 1.4 决策考古（最关键步骤）
回溯整个对话历史，主动提取：
1. 关键设计决策：选了哪个方案、考虑过哪些备选、每个被否决的具体原因
2. 明确否决的方案：逐条列出，附带否决原因
3. 用户偏好与隐式约定：用户表达的偏好（如"不要用 X 库"），即使没写入文档也必须收集
4. 遇到的坑和 workaround：底层库 Bug、兼容性问题、特殊 workaround

## 阶段 2：生成交接文档

在 `.kiro/context/` 目录创建或覆盖 `handoff.md`：

```markdown
# 🤖 第 N 期交接文档

> 版本: vX.Y.Z | 更新时间: YYYY-MM-DD

## 🎯 系统当前状态
### 基线数据
- 测试总数/通过/失败
- 版本号
- 测试命令 / 开发服务器命令

## 🔥 当前正在进行的工作
### 目标
### 已完成
### 中断点（精确到文件和行号）

## 🧠 关键设计决策与推理链
> 新 AI 必须严格遵守，不得擅自推翻。

### 决策 1: [标题]
- 选择: [最终方案]
- 备选方案: [其他方案]
- 否决原因: [为什么不选]

## 🚫 明确否决的方案
> 新 AI 严禁重新提出，除非用户主动要求重新评估。
1. [方案名] — 否决原因

## 📋 用户偏好与隐式约定
1. [偏好/约定内容]

## 🧪 质量与架构防线
- 测试状态 / 遗留报错 / DSP 图谱状态

## ⚠️ 避坑指南
踩过的坑、底层库 Bug、workaround

## ⏭️ 新 AI 的第一步任务
1. 阅读本交接文档
2. 先复述对当前设计方案和约束的理解，等待用户确认
3. [具体下一步行动]
```

## 阶段 3：交接仪式

生成文档后提示用户：

> 上下文接力棒已备好。请开新的聊天窗口，然后说：
> "请读取 `.kiro/context/handoff.md`，继承上下文，继续工作。"
>
> 建议先要求 AI："先复述你对当前设计方案、关键约束和否决方案的理解，我确认后你再动手。"
