# 回归守卫 (Regression Guard)

> 迁移自 GitHub 最新版 `skills/regression-guard/SKILL.md`。通过 `#regression-guard` 在聊天中手动引用。
> 修改已有代码时触发，防止新功能破坏旧功能。

## 三层防护

| 层级 | 手段 | 说明 |
|------|------|------|
| 代码级 | `git diff` + 导入链分析 | 最实时，看实际改了什么 |
| 图谱级 | DSP 依赖关系查询 | 辅助参考，可能过时 |
| 测试级 | 跑测试 + 构建 | 只要覆盖到就可靠 |

## 流程

### Step 1: 变更扫描
1. `git diff --name-only` + `git diff --stat`
2. 分析每个变更文件：修改了哪些函数/组件/导出/类型？
3. 变更分类：

| 变更类型 | 风险等级 |
|---------|---------|
| 删除/重命名导出 | 🔴 高危 |
| 修改函数签名 | 🔴 高危 |
| 修改返回值结构 | 🟡 中危 |
| 修改内部逻辑 | 🟡 中危 |
| 修改样式/布局 | 🟢 低危 |
| 新增导出 | 🟢 低危 |

🔴 高危变更必须标注 `⚠️ BREAKING CHANGE` 并通知用户。

### Step 2: 影响推导
1. 导入链分析：搜索谁 import 了变更文件，递归追踪
2. DSP 图谱查询（辅助，以代码实际导入为准）
3. 输出影响清单：直接变更 → 一级受影响 → 二级受影响

### Step 3: 回归检查清单

| # | 检查项 |
|---|--------|
| 1 | 受影响模块的导入语句是否仍然有效？ |
| 2 | 受影响模块使用的 API 签名是否匹配？ |
| 3 | 受影响模块的类型定义是否兼容？ |
| 4 | 受影响模块的行为预期是否不变？ |
| 5 | 是否有对应的测试覆盖？（无则先补测试） |

### Step 4: 测试验证
1. 运行全量测试，确认通过
2. 类型与语法检查
3. 前端项目：如涉及可视区域，必须通过浏览器验证截图确认无跑版

### Step 5: 图谱同步
如涉及新建模块、修改导出、增删依赖 → 同步更新 `.dsp/`

### Step 6: 完成报告

```
## 回归守卫报告
### 变更概览
| 文件 | 变更类型 | 风险等级 |
### 影响分析
- 直接影响 / 间接影响 / 破坏性变更
### 测试验证
- 测试 / 构建 / 类型检查 结果
```

## 红旗信号——立刻停下

- "这个改动很小，应该不会影响其他地方" → 做影响分析
- "测试太慢了，先跳过" → 必须跑测试
- "这个模块没有测试，算了" → 先补测试
- "构建通过就行了" → 构建通过 ≠ 功能正常
