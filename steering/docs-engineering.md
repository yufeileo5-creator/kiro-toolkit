# 文档工程标准 (Documentation Engineering)

> 迁移自 `docs-writer`、`docs-changelog` 技能及 GEMINI.md 文档工程规则。
> 编辑 Markdown 文件时自动加载。

## 写作标准

### 语气与风格
- 使用"你"称呼读者，主动语态，现在时
- 专业、友好、直接
- 使用简单词汇，避免行话和营销用语
- 明确区分"必须"和"建议"
- 使用缩写时先写全称

### 语言与语法
- 避免拉丁缩写，用"例如"代替"e.g."
- 使用牛津逗号
- 示例中使用有意义的名称，避免 "foo"、"bar"

### 格式
- 每个标题后必须有至少一段概述文字，然后才能出现列表或子标题
- 标题使用句子大小写
- 顺序步骤用编号列表，非顺序用项目符号
- UI 元素加粗，文件名/代码/命令用代码字体
- 链接使用描述性锚文本，避免"点击这里"

### 结构
- 开头说明读者将获得什么（BLUF 原则）
- 使用层级标题支持用户旅程
- 步骤以祈使动词开头
- 条件放在指令前面（如"在设置页面，点击..."）
- 适当时以"下一步"结尾

## 变更日志规范

功能变更、Bug 修复或破坏性变更时，按 Keep a Changelog 格式更新：

```markdown
## [vX.Y.Z] - YYYY-MM-DD
### Added（新增）
- 具体内容

### Fixed（修复）
- 具体内容

### Changed（变更）
- 具体内容
```

## API 文档

所有对外暴露的函数/接口必须编写 JSDoc/TSDoc，包含参数说明、返回值、异常情况。禁止裸导出无注释的公共 API。
