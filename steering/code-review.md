# 代码审查流程 (Code Review)

> 迁移自 `code-review` 技能。通过 `#code-review` 在聊天中手动引用。

## 审查流程

### 阶段 1：准备

1. 检查 git 状态，识别变更文件：`git status --short` / `git diff --name-only`
2. 获取变更统计：`git diff --stat`
3. 确定变更范围（新功能/Bug修复/重构）

### 阶段 2：多维度审查

从以下 6 个维度并行审查所有变更：

| 审查维度 | 关注点 |
|---------|--------|
| 安全审计 | 安全漏洞、注入风险、敏感信息泄露 |
| Bug 猎手 | 逻辑错误、静默失败、边界条件 |
| 代码质量 | 可维护性、简洁性、项目规范一致性 |
| 契约审查 | 类型设计、API 变更、数据建模 |
| 测试覆盖 | 测试质量和完整性（如有测试文件变更） |
| 历史上下文 | git blame、文件修改历史、相关提交 |

### 阶段 3：置信度评分

对每个发现的问题评分（0-100）：
- 0：误报，经不起推敲
- 25：可能是问题，但无法验证
- 50：确认是问题，但可能是吹毛求疵
- 75：高度确信，会在实际中触发
- 100：绝对确定，频繁发生

只保留评分 ≥ 80 的问题。

### 误报排除

以下不算问题：
- 未变更代码中的既有问题
- Linter/类型检查器会捕获的问题
- 通用代码质量建议（除非项目规范明确要求）
- 明显是有意为之的功能变更

## 输出格式

```
# 📋 代码审查报告

## 🎯 质量评估
- 质量门控：✅ 可提交 / ❌ 需修复
- 阻塞问题数：X

## 🚫 提交前必须修复
1. [文件:行号] 问题描述 — 影响

## ⚠️ 建议修复
1. [文件:行号] 问题描述

## ✨ 改进建议
1. 改进描述 — 优先级/影响
```
